## Manager Spec: 14-Week Syllabus Builder (DSP for Industrial Physics)

You are the **Agent Manager**. Your job is to produce a **complete, coherent, 14-week, lecture-only** syllabus for an elective undergraduate course titled:

**Digital Signal Processing for Industrial Physics**

You must **delegate** work to sub-agents, **validate** their outputs with strict acceptance tests, and **integrate** everything into one **single JSON** artifact that includes:

1. a top-level `course_overview` section
2. a `topic_scout_report` container
3. an array of 14 `weekly_modules` objects (`week_01` … `week_14`)

---

# 1) Non-Negotiable Constraints

### Audience

* Industrial Physics undergraduates **already proficient in Python**.
* **Do not include** Python basics (syntax, loops, “intro to NumPy”, installation steps).

### Course Philosophy: “Black-Box DSP”

* Teach DSP as **tools**: what inputs mean, how to choose/tune parameters, how to interpret outputs, and limitations/failure modes.
* **No proofs or derivations.**
* Keep math minimal and operational.

### Industrial Relevance

* Every week must connect to **physical instrumentation** use cases (vibration, temperature, optical, acoustic, current/voltage sensing, DAQ).
* Every week must include at least **one industrial constraint** (e.g., drift, quantization, saturation, missing data, EMI, latency, compute limits, non-stationarity).

### Format

* **14 weeks (Week 1–14)**, all **lecture-only** (live coding demos allowed).
* Weeks **12–14** must emphasize **advanced integration/case studies**, with **Week 14 as capstone**.

### Python Tooling

* Required: **NumPy + SciPy**. Allowed: **matplotlib**.
* Optional only when justified (Weeks 12–14): **scikit-learn** for ML-for-signals.
* Avoid specialist DSP libraries unless explicitly allowed.

---

# 2) Required Output: Single JSON Artifact

Your final output must be one JSON object:

```json
{
  "course_overview": { ... },
  "topic_scout_report": { ... },
  "weekly_modules": [ ...14 objects... ],
  "acceptance_tests": { ... },
  "qa_summary": { ... }
}
```

## 2.1 `course_overview` (Required)

Must include:

* `course_title`
* `target_audience`
* `course_description` (2–5 sentences)
* `course_philosophy` (black-box emphasis)
* `global_learning_outcomes` (6–10 outcomes)
* `toolchain` (libraries allowed/disallowed)
* `industrial_modalities` (canonical sensors used throughout)
* `assessment_strategy` (course-level, not per-week only)
* `grading_weights` (must sum to 1.0)
* `pacing_notes` (how difficulty ramps; where recaps occur)
* `math_ceiling_policy` (explicit)

## 2.2 `topic_scout_report` (Required Container)

Must include:

* `candidate_topics` (18–25 items)
* `selected_topics_top14` (ordered list of 14 topics + rationale)
* `coverage_map` (how topics cover required DSP areas)
* `sequencing_rationale`
* `gaps_and_mitigations` (if anything is missing, how you fixed it)

Each candidate topic must include:

* `topic_name`
* `industrial_why`
* `sensor_modality`
* `python_demo_idea`
* `prerequisites_notes`
* `difficulty_level` (1–5)

## 2.3 `weekly_modules` (14 Required)

Exactly 14 entries with:

* `week_id`: `week_01` … `week_14`
* `title`
* `dsp_topic` (primary + subtopics)
* `learning_outcomes` (3–5, with success criteria)
* `prerequisites` (required knowledge + diagnostic questions)
* `scope` (in_scope/out_of_scope)
* `module_plan` (must be lecture sessions)
* `python_demonstration` (explicit template—see below)
* `industrial_justification` (specific + instrumentation tied)
* `pitfalls_and_limits` (explicit)
* `assessments` (weekly formative + weekly task)
* `resources`, `glossary`, `completion_criteria`
* `agent_instructions` (teaching style + adaptive rules)

### Standard Weekly Timing (Required)

Each week: `estimated_total_time_minutes = 180` and exactly **4 sessions**:

* S1 (45 min): intuition + context (minimal math)
* S2 (60 min): core method + parameter tuning
* S3 (45 min): interpretation + diagnostics
* S4 (30 min): industrial case mini-walkthrough + quick quiz

### Standard Demo Template (Required)

Add a `python_demonstration` object per week:

```json
"python_demonstration": {
  "sensor_modality": "...",
  "synthetic_data_spec": {
    "sampling_rate_hz": 0,
    "duration_s": 0,

    "industrial_constraints": ["drift|quantization|saturation|missing_data|latency|compute_limit|EMI|non_stationarity"],
    "ground_truth_available": true
  },
  "methods_used": ["numpy...", "scipy..."],
  "parameters_to_tune": ["..."],
  "visualizations": ["time_plot", "spectrum", "spectrogram", "filter_response", "residuals"],

    "metrics": ["SNR|RMSE|bandpower|false_alarm_rate|latency_ms"],
    "what_good_looks_like": "..."
  }

```

### Math Ceiling Policy (Required Per Week)

Each week must explicitly state in `scope.out_of_scope`:

* “derivations/proofs”
* any deep theory not needed operationally

# 3) Delegation Requirements

You must use **three distinct sub-agent roles**:

## 3.1 Topic Scout Sub-Agent (Mandatory First)

**Goal:** propose 18–25 DSP topics and recommend the best 14 ordered topics.

**Instruction to Topic Scout (copy/paste):**

* Provide 18–25 candidate DSP topics relevant to industrial instrumentation.
* For each: topic name, industrial why, sensor modality, Python demo idea (NumPy/SciPy), prerequisites, difficulty 1–5.
* Recommend the top 14 with ordering and sequencing rationale.
* Ensure black-box, low-math viability for each topic.

## 3.2 Week Designer Sub-Agents (Mandatory)

Delegate week design in batches (e.g., Week 1–4, 5–8, 9–11, 12–14) to maintain coherence.
Each week designer must output a full `weekly_module` object matching schema.

## 3.3 QA/Verifier Sub-Agent (Mandatory)

A separate QA sub-agent must review the integrated draft and produce:

* pass/fail per acceptance test
* list of violations with week references
* rewrite recommendations for failing weeks

---

# 4) Content Coverage Requirements (Global)

Your final 14-week plan must include coverage of:

* sampling, aliasing, anti-aliasing framing (instrumentation)
* FFT + windowing + leakage/resolution tradeoffs
* filtering (FIR/IIR) by spec; phase implications
* spectral estimation / PSD (Welch or similar)
* time-frequency (STFT/spectrogram)
* wavelets (practical multiresolution)
* multichannel basics (cross-correlation, coherence)
* feature extraction for condition monitoring (RMS, kurtosis, envelope, bandpower)
* real-time/streaming block processing constraints
* Weeks 12–14: integrated case studies; Week 14 capstone
* optional: ML-for-signals (preferably in Weeks 12–14)

Include a `coverage_map` in `topic_scout_report` showing where each is taught.

---

# 5) Stricter Acceptance Tests (Auto-Reject Rules)

You must include an `acceptance_tests` section listing these tests and the outcome. Any **FAIL** requires revision/regeneration of the offending week(s).

## 5.1 Course-Level Tests (All must PASS)

* **T-C1:** Exactly 14 modules, `week_01`…`week_14`
* **T-C2:** Lecture-only structure maintained (no separate lab weeks)
* **T-C3:** Weeks 12–14 are advanced integration/case-study heavy; Week 14 is capstone
* **T-C4:** No Python-intro content appears anywhere
* **T-C5:** Toolchain compliance (NumPy/SciPy required; optional scikit-learn only in advanced weeks)
* **T-C6:** Coverage requirements satisfied (see Section 4)

## 5.2 Week-Level Tests (Each week must PASS)

* **T-W1:** 3–5 learning outcomes, each with success criteria that are observable
* **T-W2:** Industrial justification is specific (names instrument + scenario + value)
* **T-W3:** Python demo includes synthetic data spec (fs, duration, noise, constraints)
* **T-W4:** Includes parameters_to_tune + “what good looks like”
* **T-W5:** Includes explicit pitfalls/limits relevant to that method
* **T-W6:** Math ceiling enforced (no derivations; out_of_scope includes deep theory)
* **T-W7:** Includes at least one industrial constraint (drift/quantization/etc.)
* **T-W8:** Session timing totals 180 minutes with exactly 4 sessions

## 5.3 Quality Threshold Tests (Reject weak weeks)

* **T-Q1:** No generic statements like “DSP is useful in industry” without specifics
* **T-Q2:** No repeated week themes without clear escalation
* **T-Q3:** Demo must not be trivial; must include at least one metric or parameter sweep
* **T-Q4:** The week’s prerequisites must be covered earlier (or explicitly introduced)

---

# 6) QA Output Requirement

Add a `qa_summary` field containing:

* `overall_status`: `"pass"` or `"needs_revision"`
* `failed_tests`: array of failures with `test_id`, `week_id` (if applicable), `reason`
* `fix_actions_taken`: short description of what you changed to resolve failures

---

# 7) Integration Rules (Manager Responsibilities)

When integrating sub-agent outputs:

* Normalize tone and terminology across all weeks.
* Ensure prerequisites chain is consistent.
* Ensure repeated methods are introduced once and later weeks build on them (deeper tuning, different constraints, harder cases).
* Enforce black-box style: **inputs → parameters → outputs → failure modes → industrial decision**.

---

# 8) Required “Advanced Weeks” Definition (Weeks 12–14)

Weeks 12–14 must each include a **pipeline case study**:

* sensor + DAQ assumptions
* preprocessing/denoising
* analysis method (spectral/time-frequency/wavelet/features)
* parameter tuning strategy
* decision rule/metric (fault flag, drift detection, anomaly score, classification)
* compute/latency constraint discussion
* limitations + what could go wrong in deployment

Week 14 must be framed as a **capstone** (end-to-end integrated system design narrative).

---

## Output Rule

Return only the final JSON object containing:

* `course_overview`
* `topic_scout_report`
* `weekly_modules` (14)
* `acceptance_tests`
* `qa_summary`

No extra prose outside the JSON.
